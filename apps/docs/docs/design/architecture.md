# 系统架构

Forge Studio 的核心架构采用了**“生与产分离”**（Producer-Consumer）模式。我们将系统清晰地划分为两个平行的宇宙：`@forge/editor` 和 `@forge/core`。它们之间通过严格的 JSON 场景数据协议进行通信。

## 1. 为什么需要生与产分离？

传统的 3D 编辑器往往将编辑态代码（如辅助网格、坐标变换工具、UI 面板）与运行态代码（渲染逻辑、动画循环）深度耦合。这会导致：
- **产物体积臃肿**：无论最终业务场景是否需要，编辑器相关的代码都会被打包进去。
- **性能损耗**：编辑器为了实现复杂的交互，往往会牺牲一部分运行效率。
- **维护困难**：UI 逻辑与渲染逻辑交织，导致项目后期难以迭代。

为了解决这些痛点，我们设计了两套完全独立的运行时环境。

## 2. 两个平行宇宙

### 🪐 生产者：@forge/editor

`@forge/editor` 是一个功能大而全的 3D 可视化 IDE。

- **定位**：提供极致的开发者体验 (DX) 和丰富的 UI 交互。
- **核心职能**：场景搭建、资产管理、材质调节、动画编排。
- **设计原则**：不限制体积，尽一切可能提升编辑效率。内置全套辅助工具，如 `TransformControls`、网格、幽灵球吸附、历史记录等。
- **输出**：在完成编辑后，它将导出一份标准化的 JSON 场景描述文件。

### 🪐 消费者：@forge/core

`@forge/core` 是一个纯粹、极致轻量化的渲染引擎。

- **定位**：专注于高性能渲染，专供最终的业务大屏或看板使用。
- **核心职能**：解析 JSON 协议数据，精准还原场景。
- **设计原则**：追求极致轻量化，完全剔除任何 UI 代码和编辑态辅助代码。支持按需加载（Tree-shaking），如果场景中没有灯光，则灯光相关的渲染模块不会被打包。
- **输入**：接收标准的 JSON 场景描述文件，并提供简洁的 API 供业务端进行数据驱动和事件监听。

## 3. 项目工程目录结构

基于上述架构理念，我们的 Monorepo 工程目录结构设计如下：

```text
three-forge-studio/
├── apps/                          # 应用层：对外交付的具体产品或集成环境
│   ├── demo/                      # 演示沙箱应用。完整集成 Editor 和 Core，验证“编辑->协议->渲染”闭环
│   └── docs/                      # 官方文档系统，基于 VitePress 构建
├── packages/                      # 核心包层：独立、解耦的 NPM 模块
│   ├── editor/                    # @forge/editor (生产者)：功能大而全的 3D IDE，基于 Vue 3 + Element Plus
│   ├── core/                      # @forge/core (消费者)：极度轻量化的 Vanilla TypeScript 运行时渲染引擎
│   ├── utils/ (规划中)            # @forge/utils (公共工具)：存放数学计算、颜色转换、UUID生成等与视图无关的纯函数
│   └── types/ (规划中)            # @forge/types (协议标准)：存放两个宇宙唯一沟通的 JSON Schema 和 TS 类型定义
├── pnpm-workspace.yaml            # Monorepo 工作区配置，定义 apps 和 packages 的关联
└── package.json                   # 根级包描述文件，定义全局依赖与跨包的统一构建脚本
```

### 目录职责详述

*   **`apps/demo/`**: 这是架构设计的“试验田”。它不仅用于本地开发调试，更是向外展示如何将 `@forge/editor` 嵌入到 Web 应用中，并如何通过点击“预览”按钮将数据喂给 `@forge/core` 进行最终渲染。
*   **`packages/editor/`**: 包含所有的 UI 面板（属性栏、场景树）、编辑态辅助工具（TransformControls、GridHelper）、用户交互记录（撤销/重做）。体积不受限，追求极佳的 DX（开发者体验）。
*   **`packages/core/`**: 纯净的运行时。只包含解析纯函数（如 `createMaterialFromJSON`）和轻量级的 Three.js 场景构建逻辑。设计目标是能被业务端完美 Tree-shaking。
*   **`packages/utils/`**: 存放**业务无关、视图无关**的公共基础方法库（如节流防抖、深度拷贝、矩阵/向量辅助计算等）。`editor` 和 `core` 都可以安全地依赖它，而不用担心互相耦合。
*   **`packages/types/`**: 架构中的“宪法”。只要涉及数据结构的变更，必须先在这里修改 Schema，从而确保 Editor 导出的数据绝对能被 Core 解析。

## 4. 核心子系统规划 (Core Subsystems)

在宏观的“生与产分离”骨架之上，为了达到顶级工业标准，我们的架构中还预留了以下四大核心子系统的位置。它们是支撑整个引擎顺畅运转的“血肉”：

### 1. 资产管理器与缓存系统 (Asset Manager) -> 归属 `@forge/core`
JSON 协议中记录的只是贴图和模型的 URL。为了防止内存溢出 (OOM) 和重复下载，`@forge/core` 内部必须拥有一个全局单例的资源调度中心。
- **职责**：拦截所有外部资源的加载请求，提供统一的内存缓存 (Cache)、去重、释放 (Dispose) 机制，并向外派发统一的加载进度事件 (Loading Progress)。

### 2. 运行时交互总线 (Runtime Event Bus) -> 归属 `@forge/core`
为了彻底打通 3D 世界与外部 2D 业务 DOM 的通信。
- **职责**：内置高效的射线拾取 (Raycaster) 轮询机制。自动拦截 Canvas 的鼠标/触摸事件，并将其转化为简单的 API（如 `core.on('click', node)`），让业务开发者可以零门槛实现“点击 3D 模型弹出 HTML 详情页”等需求。

### 3. 命令模式与撤销重做 (Command System) -> 归属 `@forge/editor`
现代可视化编辑器的基石。
- **职责**：在 Editor 中禁止直接修改场景状态，所有针对场景树的操作（如移动位置、修改材质颜色）必须被封装为具体的 `Command` 对象。每个对象必须实现 `execute()` 和 `undo()` 方法，并由全局的 History 栈管理，从而实现无限步的撤销与重做。

### 4. 通用补间与时间轴系统 (Tween & Timeline) -> 归属 `@forge/core` 基础库
除了传统的骨骼动画，业务中充斥着大量的数值过渡需求（如摄像机巡游漫游、颜色呼吸灯）。
- **职责**：在不引入庞大动画库的前提下，内部集成极度轻量的数字补间（Tween）逻辑。通过解析 JSON 中的特定 `animations` 节点，接管数值随时间的平滑过渡插值。

## 5. 沟通的桥梁：JSON 场景协议

在这两个宇宙之间，唯一的契约就是 JSON 场景数据协议。

- `@forge/editor` 负责**生产** JSON。
- `@forge/core` 负责**消费** JSON。

这种高度解耦的设计，使得 Core 不关心 JSON 是谁生成的，Editor 也不关心 JSON 最终在哪里被渲染。未来，我们甚至可以用其他的底层技术栈（如纯 WebGPU，甚至是服务器端渲染）来消费这份 JSON 数据，而完全不需要重写整个编辑器。

## 6. 保证“所见即所得”与系统整合

如果 Editor 和 Core 的场景各自完全独立，如何保证用户在 Editor 中看到的效果与最终在 Core 中运行的效果（WYSIWYG）完全一致？

我们采用以下策略来解决这个问题：

### 策略 A：共享核心解析原子函数 (Shared Utilities)

虽然场景各自独立，但**“数据如何映射为 Three.js 对象”**的逻辑必须是同一套。

我们将这些映射函数抽象为纯函数，抽离到 `@forge/core` 中作为底层工具库导出（未来也可以根据需要下沉到 `@forge/utils` 或 `@forge/types` 中），例如：
- `createMaterialFromJSON(json)`
- `createLightFromJSON(json)`

`@forge/editor` 在其内部庞大且“脏”的编辑场景中，仍然使用这套底层的共享函数来生成物体的材质、网格和光影。这就从物理层面上保证了，两边对于同一个 JSON 描述，构建出来的 WebGL 渲染管线参数是绝对一致的。

### Demo 整合与沙箱预览

为了将上述所有理念串联并验证，我们在 `apps/demo` 下提供了一个完整的集成环境。在这个 Demo 中：
1. 我们会运行完整的 `@forge/editor` 实例供用户进行 3D 编辑。
2. 同时，系统会提供一个“预览模式”（PreviewSandbox）。
3. 当用户进入预览时，Editor 将当前状态导出为 JSON 协议数据，并注入到一个由纯 `@forge/core` 驱动的隔离沙箱（Canvas）中运行。

通过这个集成的 Demo 应用，开发者可以清晰地验证“生与产分离”的架构下，数据是如何在两个平行的宇宙之间精准无误地流动的。
